# Lottery Terminal Operations System

## What This Is

Локальная веб-система для работы с лотерейными билетами через клиентские компьютеры и один главный терминал исполнения. Пользователь выбирает лотерею, проходит авторизацию, видит актуальные тиражи, формирует билет, подтверждает покупку и получает статусы покупки и проверки билета; администратор управляет доступностью лотерей, очередью, ошибками и состоянием терминала.

## Core Value

Каждая подтвержденная покупка должна предсказуемо пройти путь от веб-интерфейса до единственного главного терминала с корректным резервированием/списанием денег, понятным статусом и полным журналом событий.

## Current Milestone: v1.1 Domain Consolidation and Truth Model

**Goal:** Collapse split runtime truth into canonical `purchase` and `draw` models without breaking the working Big 8 contour.

**Target features:**
- canonical `purchase` aggregate with separate execution, result, and visibility axes
- canonical `draw` lifecycle with explicit `open -> closed -> settled` control
- additive storage migration with compatibility read models for current web/admin surfaces
- durable `purchase_attempt` history plus worker/lock hardening before transport replacement

### Active

- [ ] Внутренняя система баланса корректно резервирует, списывает, возвращает и начисляет средства по жизненному циклу заявки и билета.
- [ ] Администратор управляет доступностью лотерей, очередью, приоритетными заявками, ошибками и состоянием терминала.
- [ ] После розыгрыша система автоматически получает результат билета через главный терминал и обновляет баланс пользователя.
- [ ] Кодовая база остается модульной, поэтапно проверяемой и полноценно документированной для последующих изменений.

### Out of Scope

- Поддержка нескольких одновременно активных главных терминалов — противоречит текущему ограничению системы на одну точку исполнения.
- Прямой доступ пользователя к браузеру или автоматизации главного терминала — ломает модель централизованного контроля.
- Генерация произвольного исполняемого кода из пользовательских параметров билета — обработчики должны быть заранее определены и проверяемы.
- Публичные интернет-сценарии пополнения/вывода средств — текущее ТЗ описывает только внутренний баланс системы.
- Нативные мобильные приложения — в рамках v1 нужен единый веб-интерфейс для локальной сети.

## Context

- Проект зеленого поля: в рабочей директории нет существующей кодовой базы, проект инициализируется с нуля через GSD.
- Базовый домен: пользователи работают за клиентскими ПК в локальной сети, а фактическое исполнение покупки и считывание результатов сосредоточены на одном главном терминале.
- Домен жестко требует разделения общего каркаса и лотерейно-специфичной логики: интерфейс общий, обработчики покупки/результатов — отдельные модули.
- Финансовая часть не внешняя, а внутренняя: нужен надежный журнал движения средств, резервирования, списания, возврата и начисления выигрыша.
- Пользователь прямо потребовал поэтапную и помодульную реализацию, чтобы каждый кусок можно было запускать и проверять частично, не дожидаясь всей системы.
- Для поддержки будущих изменений документация считается не “nice to have”, а обязательной частью результата: архитектура, модули, контракты, тестовые сценарии и runbook'и должны жить рядом с кодом.

## Constraints

- **Topology**: Один главный терминал — система обязана исключать параллельное активное исполнение покупок.
- **Network**: Локальная сеть — пользователь работает только через веб-интерфейс, без прямого доступа к терминалу.
- **Automation**: Только заранее подготовленные обработчики по каждой лотерее — никаких произвольных сценариев из пользовательского ввода.
- **Reliability**: Заявки, тиражи, билеты и баланс должны переживать ошибки, повторные попытки и перезапуски без потери состояния.
- **Financial Integrity**: Любая операция с деньгами должна быть трассируема и идемпотентна по отношению к заявке или билету.
- **Delivery Model**: Реализация должна быть фазовой, модульной и пригодной для частичного smoke/integration тестирования на каждом шаге.
- **Documentation**: Архитектурные решения, границы модулей, расширение новыми лотереями и сценарии проверки должны быть зафиксированы письменно.
- **Technology Choice**: Конкретный стек не задан ТЗ и должен быть выбран в первой фазе на основании модульности, транзакционной надежности и удобства тестирования.

## Key Decisions

| Decision | Rationale | Outcome |
|----------|-----------|---------|
| Строить систему как общий каркас с подключаемыми lottery adapters | Это удерживает единый UX и позволяет добавлять/менять лотереи без переписывания ядра | — Pending |
| Резать проект на мелкие вертикальные фазы с частичной проверкой | Ты прямо требуешь поэтапную реализацию, которую можно запускать и проверять кусками | — Pending |
| Вести баланс через внутренний ledger с резервом как отдельным состоянием | Без этого нельзя надежно доказать корректность списаний, возвратов и начислений | — Pending |
| Изолировать терминал за очередью и worker-слоем | Это снимает гонки между клиентами и дает одну точку контроля над браузерной автоматизацией | — Pending |
| Фиксировать архитектуру, модули и тестовые ходы как артефакты GSD рядом с кодом | Поддерживаемость и последующие изменения здесь критичны, а не факультативны | — Pending |
