# Milestone v1.2: UI Contract, Readability, and Closure Truth

**Status:** COMPLETED 2026-04-25
**Phases:** 26-34
**Focus:** user UI cleanup, Russian-first statuses, admin readability, draw-close one-step settlement, 5-second draw sync, admin balance control, runtime reset truth, detached reserve reconciliation, client result push, and honest reserve/debit/winnings behavior

## Overview

This milestone turns the already working lottery contour into a clean operator/client surface without lying about runtime state.

The main rule for this milestone is strict: UI changes are not accepted as "done" unless the underlying mechanics match the new interface. If a label, status, or action changes, the backing read model, ledger behavior, settlement flow, or notification path must be corrected in the same phase.

## Locked Decisions

1. User and admin interfaces must show Russian statuses only; raw internal states such as `success`, `confirmed`, or `canonical result: win` are not acceptable in the visible UI.
2. Draw availability on the user purchase screen must refresh automatically; no extra manual action should be needed to see the current draw.
3. Low-value helper text, technical status copy, and service-only fields must be removed from the visible cashier UI instead of being left as clutter.
4. After terminal purchase succeeds, money must leave reserve and become a final debit; after a winning result, the winning amount must be credited correctly and audibly traceable in the ledger.
5. Draw closure on the admin side must become a one-step contour: close draw -> publish result -> client receives updated result/status without an extra manual settlement action.
6. The client purchase page should show only the information needed to choose a draw, compose tickets, confirm price, and see wallet state relevant to the current purchase.
7. The user cabinet must stop exposing internal identifiers and raw backend result strings where they do not help the cashier.
8. Broken encoding in admin surfaces is a data-quality bug, not a styling issue; it must be treated as incorrect text output and fully removed from touched contours.
9. If a purchased ticket already exists before draw closure, both user and admin surfaces must show that it is purchased instead of leaving it in a pre-purchase style state.
10. Any UI action removed from the interface must have its backing mechanic removed or collapsed too; dead operator steps are not allowed to remain behind the scenes.

## Detailed Change Contract

### User Purchase Surface

1. The draw selector must keep itself current without a manual reload button or manual poke to refresh the list.
2. Ticket-count controls must remain compact and stable as the ticket count changes.
3. Remove purchase-page text that only explains technical polling, service state, or obvious operator instructions once the UI itself is clear enough.
4. Remove low-value wallet/draw helper rows that do not help the cashier complete the current purchase.
5. Keep only the wallet and draw facts that matter to the purchase decision: available funds, reserved funds, current draw, and draw time if still relevant after cleanup.
6. Recheck reserve lifecycle: after terminal purchase success, the request must no longer leave stale money in reserve.
7. Recheck winnings lifecycle: after result publication, credited winnings must reach the visible balance and the ledger path must stay auditable.

### User Cabinet Surface

1. Translate all request, ticket, result, and claim statuses into Russian.
2. Remove raw backend values such as internal result strings and other technical phrasing from visible tables.
3. Remove explanatory subtitles and helper text that add noise but no operating value.
4. Hide or collapse internal request identifiers if they are not needed for the cashier workflow.
5. Keep live updates intact while cleaning the view.

### Admin Surface

1. Fix all visible mojibake and broken encoding in the receiver/overview/admin status blocks touched by this milestone.
2. Make draw creation layout stable so date/time input never obscures the primary action button.
3. Show truthful pre-close ticket state: a purchased ticket must already read as purchased before result publication.
4. Collapse draw closure into one final operator action; separate post-close settlement publication must disappear.
5. Result propagation to the client must happen immediately after draw close, without extra manual steps.
