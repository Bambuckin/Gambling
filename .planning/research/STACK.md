# Stack Research: Lottery Terminal Operations System

## Recommended Baseline

Конкретный стек еще не зафиксирован, но форма системы жестко подталкивает к такому техническому профилю:

1. Единый типизированный язык для backend, web UI и terminal integration layer, чтобы не плодить лишние границы перевода моделей.
2. Транзакционная реляционная база данных для пользователей, баланса, заявок, билетов, журналов и реестра лотерей.
3. Отдельный durable queue/worker слой для покупки, повторных попыток, обновления тиражей и сверки билетов.
4. Browser automation adapter, спрятанный за интерфейсом обработчика лотереи, а не размазанный по бизнес-логике.
5. Набор тестов трех уровней: unit для правил и стоимостей, integration для orchestration/ledger/queue, e2e-smoke для основных пользовательских путей и terminal stub.

## Selection Criteria

| Area | Needed Property | Why It Matters |
|------|-----------------|----------------|
| Backend | Сильная типизация и четкие модульные границы | Здесь много состояний заявки и денег, ошибки дорого стоят |
| Persistence | Транзакции, блокировки, аудируемость | Баланс и очередь нельзя держать на “авось” |
| Queue | Надежная последовательная обработка и retry semantics | Главный терминал один, гонки недопустимы |
| UI | Общий каркас + динамические формы | Все лотереи живут в одном интерфейсе, но с разными параметрами |
| Automation | Детеминированные сценарии и устойчивые селекторы | Иначе terminal layer превратится в хрупкую кашу |
| Testing | Изолируемые адаптеры и stubs | Ты хочешь проверять систему частями, а не только целиком |
| Docs | ADR, модульные README, runbook'и | Иначе изменять систему потом будет дорого и тупо больно |

## Recommendation

Первая фаза должна закончиться не просто “выбором стека”, а зафиксированным ADR-пакетом:

- приложение и UI;
- слой данных и миграций;
- очередь и планировщики;
- контракт терминального адаптера;
- стратегия тестирования по уровням;
- правила документации для каждого модуля.

Это даст не абстрактный “выбор технологий”, а основу, которую потом не придется рвать по живому.
