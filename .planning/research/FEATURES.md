# Feature Research: Lottery Terminal Operations System

## Table Stakes

| Category | Feature Set | Complexity | Why It Is Table Stakes |
|----------|-------------|------------|-------------------------|
| Access | Логин, сессия, возврат в выбранную лотерею, роли | Medium | Без этого пользователь теряет контекст и нельзя отделить администратора |
| Lottery UX | Каталог лотерей, единый shell, lottery-specific формы | Medium | Это основная поверхность продукта |
| Draw Data | Актуальные тиражи, freshness, запрет покупки по устаревшим данным | Medium | Пользователь должен видеть актуальные основания для покупки |
| Purchase Flow | Валидация, расчет цены, подтверждение, статусы заявки | High | Это центральный пользовательский сценарий |
| Queue/Terminal | Одна активная операция, очередность, retry, результат из браузера | High | Это главный технологический узел системы |
| Balance | Резерв, списание, возврат, ledger, начисление выигрыша | High | Без надежной финансовой модели система бессмысленна |
| Ticket Results | Хранение билета, сверка после розыгрыша, обновление баланса | High | Пользователь должен получить финальный исход |
| Admin/Ops | Управление лотереями, очередь, ошибки, терминал, приоритет | High | Иначе система останется неуправляемой в эксплуатации |

## Differentiators Worth Deliberate Design

| Category | Feature | Why It Matters |
|----------|---------|----------------|
| Maintainability | Lottery adapter contracts и расширяемый registry | Позволяет добавлять новую лотерею без перелома ядра |
| Reliability | Идемпотентность финансовых и terminal операций | Спасает от двойных списаний и дрейфа состояний |
| Operability | Четкие журналы, предупреждения, runbook'и | Администратор должен видеть, что сломалось, не вскрывая код |
| Delivery | Частично проверяемые вертикальные срезы | Это прямое требование к способу реализации проекта |

## Anti-Features / Things To Avoid

| Feature | Why To Avoid In v1 |
|---------|--------------------|
| Несколько одновременно активных терминалов | Ломает текущее бизнес-ограничение и резко усложняет консистентность |
| Случайная генерация логики покупки из JSON/пользовательского ввода | Это прямой путь к неуправляемому поведению |
| Смешивание UI-форм и terminal automation в одном модуле | Такой код потом почти невозможно безопасно менять |
| Баланс без ledger-журнала | Ошибки станут нерасследуемыми |
| “Универсальная” одна форма без registry-описания лотерей | Добавление новой лотереи превратится в ручной хардкод по всей системе |

## Build Order Implication

Правильный порядок не по “красоте экрана”, а по рискам:

1. Сначала контракты, модульные границы и стратегия проверки.
2. Потом доступ и общий shell.
3. Потом реестр лотерей и тиражи.
4. Потом ledger и orchestration покупки.
5. Потом terminal worker и post-draw result flow.
6. Потом admin/ops и полная документация на расширение/эксплуатацию.
